The derivation of an HTML rendition of an XML document is complicated by the fact that the XML document has characters with special significance in HTML.  In addition, HTML does not preserve whitespace as written in the XML document.  It is necessary to make adjustments for preserving spaces and line breaks and also "escape" XML document characters "&", "<", and ">" so they appear literally without recognition as HTML-significant special codes.

It is also a bit complicated to convert the plaintext line-number fields to  HTML fields that render as permalinks having the line number as text. 

The entire process can be accomplished with scripts and small programs, as is also the case for creation of a baseline plaintext replica.   All of the steps can be automated.  Here, a manual procedure is used to demonstrate "by hand" what the critical steps are for any automation.

The procedure given here is a continuation of the manual procedure already-provided for creating baseline plaintext replicas.  The same XML document is used by way of example [n140504b3]..The examples begin with the Microsoft Office 2013 Excel spreadsheet that is a by-product of the plaintext replica procedure [n140504d2].

The objective is simple: Provide an HTML text that renders the same as the plaintext rendition (e.g., [n140504d1]), but with the line-number fields replaced by permalinks to those very lines.   Any further annotation is accomplished by editing the baseline HTML replica using appropriate tools.

This procedure illustrates two variations for HTML.  The simplest is a version that is preformatted and must be included in HTML <pre> elements in order to present with the original whitespace, including line breaks, as illustrated by [n140504d5].  The second is a version that is ordinary HTML for inclusion in HTML <p> (and equivalent) elements, with space and line breaks preserved using special HTML markup, as illustrated by [n140504d6].

1. Prerequisites

  • A line-numbered plaintext replica of the XML document should be available already.  It is important as a check on the work and preservation of fidelity with the HTML version.  There must be no horizontal tab (HT) codes in the lines, and line breaks should be consistent.  Here the convention is to use CR-LF code pairs for line breaks to be compatible with the platform used.  It is also the form produced by the saving of Microsoft Office Excel documents as space-delimited formatted text.
      
  • It is assumed, for the illustrative manual procedure, below, that a spreadsheet used to produce the plaintext replica is also available.  If other tools are used, it is possible to start from the plaintext replica.  There is enough detail here to determine what is required for any alternative approach.
      
  • In this manual approach, Excel is used to add more repetitive columns and save it as an intermediate plaintext file.  To convert the intermediate plaintext file to HTML, a text editor with good full-text search-and-replace functions is required.  There must be a means to find all of the line breaks in the plaintext file and insert constant HTML markup in front of them.

2. The Starting Point

  • There is an existing baseline plaintext replica of the original XML document, in accordance with the procedure for that level of replica.  The document was re-indented as necessary and individual line numbers added to the beginning of each line.  The file is also named as .txt so that it is not confused with the XML document it replicates (e.g., [n140504d1]).
      
     
      
  • If an Excel document was used to produce the baseline plaintext, that document is the starting point for the next steps here (e.g., [n140504d2])..
      

       

3. Adding More Line-Numbering Columns

  • Make a separate copy of the replica form to use for making more columns and inserted material. 
      
  • Starting with the Excel example, the basic idea is to add three new columns: add a blank column to the right of the line-number column.  Then add two empty columns to the left of the line-number column.  Now use the fill function to fill the cells to the left of line-number entries with those line-number values.
       

       
  • Now make an empty column to the left of each of the line-number columns.  This will have the line-number columns (B, D, F) surrounded by empty columns (A, C, E, and G).
       
  • In the rows having line numbers, arrange to fill the empty columns with different fixed values that have no special XML/HTML characters and that cannot be matched by anything already in the file.  For example, "(+0+)" in column A, "(+1+)" in C, "(+2+)" in E, and "(+3+)" in G. 
      

      
  • Fill the empty columns with those constant values in all of the line-numbered rows.
      
  • For the last repetitive-field step, it is useful to have columns A-G all aligned the same (e.g., aligned right).  Now shrink the width of the columns A-G so that the text is exactly contained in those columns with no text removed, abbreviated at a column edge, or intruding into an adjacent column.  The objective is to have a plaintext version of the material have no spaces at all in the text produced from columns A-G.
      

      
  • Save this file as the last preparation file that is not all plaintext, such as [n140504d3].  If there are any problems in subsequent steps, this can be an useful version to review, repair, and repeat the following steps.
      
  • However this point is reached, arrange to preserve a plaintext file that is the starting point for further modifications.  In the case of a just-saved Excel file, now save that same file as space-delimited formatted text.
       

      
  • The plaintext file achieved at this point has each row beginning with a block of line numbers and additional characters, without any spaces, separated by a single space from the text of the XML document for that line, such as [n140504d4].
      

      
  • Do not modify this file further.  Save it as a backup and a reference for any trouble-shooting that may be needed.  This is also a good milestone for the provenance of the modifications to follow.  In the example, this version is identified as the baseline-HTML-prep version.

4. Preserve the XML text in HTML 

  • Using a capable text editor, open the  baseline HTML prep version for preparation of a new baseline HTML replica.  The editor should line-wrap without introduction of new line breaks.   The editor should also be capable of holding very long lines having no place for line-wrapping, no "-" or space characters.
      
     
      
  • The next steps replace all "&", "<", and ">" characters of the original documents with forms that are treated as those literal characters in HTML, instead of the same characters as part of the HTML markup instead.  Note that none of the procedures up to this point have introduced any of those characters.  It is critical to modify any "&" characters of the HTML document first.
       

       
    The first replacement is of every "&" with "&amp;".  For these changes, no regular expression is involved and the search should work either way. 
       
  • The next replacements are of "<" by "&lt;" and of ">" by "&gt;".
      

        

       
  • Now is another good point to save the file.  There are two possible variations of HTML format that can be produced after this point.  The saved version can be used as a basis for developing either or both of them.  This is also a good check-point for any troubleshooting required to resolve difficulties that might happen in use and annotation of a replica.
       

5. Choose Between Pre-Formatted and Ordinary HTML

  • If the HTML is going to be used as pre-formatted text (in HTML <pre> elements), the white space, including line breaks of the plaintext file can be left "as is."  A <pre> element will preserve that text exactly as it appears in the plaintext file. 
      
  • It is also useful to make one baseline of each kind.   That is done with the XML document being used as an example [n140504b3].
      
  • To produce a pre-formatted version, skip these steps and continue with creation of the permalink line numbers.
       
  • To produce ordinary HTML, it is necessary to overcome the HTML collapse of whitespace sequences to single spaces.   There are many variations that may be desired.  In this illustration, all whitespace is preserved for literal treatment in the HTML.  It is possible to introduce further variations by further editing after the baseline HTML replica is established.
      

6. Preserve White Space for Ordinary HTML

  • Continue with a copy of the plaintext being worked on by replacing every literal space character in the file using a regular expression search.
      

       
    Literal search for the space character code is employed to avoid accidentally altering line-break codes and forcing the document into a single line.  The original line breaks should be preserved, even though the individual lines no longer have any ordinary space characters in them.
       

      
  • The line breaks that remain in the file are useful when working with the HTML code.  However, the line breaks are ignored when the HTML is presented, as in a browser.   To have the original line breaks be presented, it is necessary to make another regular-expression search and replace.
      

       
    In this regular-expression search, the end of each line (signified by "$") is the place where the replacement will be inserted.  The line break also remains.  Different editors may have other ways of expressing this change.  The result is easily confirmed, although the space in "<br />" may lead to unfamiliar line-wrapping.  There is no problem with that.
        

7. Create the Permalinks

  • For both pre-formatted and ordinary HTML formatting, the final search and replace operations are the same.  These produce the permalink line-number anchors. For line-number nnnn, the result is going to be <a name="Lnnnn" href="#Lnnnn">nnnn</a>.
      
  • Now replace every occurrence of "(0+0)" (or whatever you used) with "<a name="L".  Notice that there is a quotation symbol in the replacement text.
       

      
  • Replace every occurrence of "(+1+)" with "" href="#L".  Here there are two quotation symbols in the replacement text.
      

       
  • Replace every occurrence of "(+2+)" with "">".
       

      
  • Replace every "(+3+)" occurrence with "</a>".
       

      
  • The pre-formatted lines will have the original formatting and the HTML formatting will have the necessary line-number markup, as in [n140504d5].
       


    This version can be saved as a baseline HTML pre-formatted replica.  Extractions and any additional editing can be accomplished with derivatives of this form.
       
  • The ordinary HTML lines will be similar to the pre-formatted version except that spaces and line-breaks of the original are replaced by HTML codes that preserve the spaces and line-ends in presentation of the ordinary HTML as in [n140504d6].  Inspection of the HTML code in a text editor may now show line-wrapping at any of the spaces that have been introduced along with the inserted HTML tags.
       

8. Wrapping-Up the HTML

  • For the pre-formatted form, selected lines, or the complete text, the amount included into an HTML document should be contained in an HTML <pre> element.
      
  • For inclusions of the ordinary form, the amount included into an HTML document will usually be contained in an HTML <p> element.
      
  • If the same lines are included at different places in a single HTML document, there will be duplication of fragment identifiers.  In general, the permalink anchor name attribute values must be unique in the HTML document in which those anchors appear.  It may be necessary to change the text in the permalink <a>-element name and href attributes to avoid name collisions in re-use of the replica material..
      
  • The baseline replicas should be preserved intact for reuse in further customizations that might be desired.
