  2014-07-09 RighteousOffice might also have hypothetical use of whole-package signing and encryption.  This probably does not have to figure in SCT, but the use of embedded signed packages might be superior in many respects.
  2014-07-07 Create a position paper on SCT and the challenge.  There is a July 15 deadline.
  2014-07-07 In addition to recalculation changes, this also applies for any refreshing that causes automatic updates (such as date-time stamps and random-number generations).  So the identification of a signed and preserved manifestation must be clear, and it must be clear what happens if a document is saved after any change.  It must be easy to see the changes.
  2014-07-07 If BlueSkyOffice (.blu) is used for RCT, we need something different for SCT illustration.  It probably doesn't start with S, because of StarOffice.  I am choosing RighteousOffice (.rto) for this strong arrangement.
  2014-07-07 A consequence for turnaround documents, including forms and spreadsheets, is that the entries and derived results are captured apart from the core document, which may be both protected and signed.  It is critical that viewing documents presents them as-signed, and any recalculate differences show up a tracked changes for review and if committed (re-signed or not).  It is always clear what is being watched.
  2014-07-07 A critical quality of SCT's signature preservation is that someone who opens a document for viewing will see it as read-only and in the form that it was signed.  This means that all entered values and computation-derived values are preserved as committed with signature.  It also means that a recalculate will track any changes that result, by default.
  2014-07-07 A critical quality of SCT is that it is signature preserving.  That is, changes are external to the signed document that is being changed and the combinatiion can then be signed, and so on.  This is important for negotiated documents, collaborative changes, and turnaround documents of many kinds.
  2014-07-06 #72.51 The ability to save local commits and also retain conflicts seems like an interesting problem, especially with various contributors going off-line, coming back on, etc.  Is this different that a cache?
  2014-07-06 #72.51 Is distributed collaboration a form of commit synchronization, even though the various document instances are "live."  It looks like decentralized synchronization and Lamport's approach need to be understood.
  2014-07-06 #72.51 It appears that SCT is more appropriate for collaboration, since something like Git collision-resolution can be applied to the externally-tracked changes.
