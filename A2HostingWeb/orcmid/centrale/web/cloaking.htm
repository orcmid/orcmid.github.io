There are many domain names used as part of Centrale's Web presence.  An anchor site, such as nfocentrale.net,  can carry site images for more than one domain.  Individual domains are hosted by web forwarding.  Web forwarding causes accesses to a given domain to be routed to a specified place on the anchor site.  However, this means that URLs that are presented to browsers will be with the anchor-site (nfoCentrale.net) domain name, not that of the anchored site.  One approach to using the intended name for the site and for bookmarking is cloaking.   Web cloaking causes the pages that are delivered to appear to be located at the domain name, and not at the anchor site.

The cloaking process is imperfect.  There are adjustments that must be made on a site so that the cloaking works well.  The key thing is that the cloaking should not confuse users of the site.

After 18 months of experimentation with cloaking, I found it unacceptable, and cloaking has been abandoned.  However, the accomodation process is still used because it is always the case that navigation may arrive via a framed page.   So the current practice is now,

  • Prepare all site pages as if cloaking is happening, so that any framed entry to the site will behave properly
  • Use web forwarding to enter hosted Centrale sites and do not make use of cloaking as a way to preserve a domain name during navigation

1.  Minimal Cloaking Accommodation

When minimal cloaking accommodations have been made, a site can be accessed via cloaking and operation is basically smooth.  If you access a domain that has minimal cloaking accommodation in place, you will notice the following:

  1. entry to the site (e.g., orcmid.com) will lead to all pages of the site being presented as if all at the address http://orcmid.com/. 
  2. exiting to a different site beyond the Centrale complex will properly change to the presentation of that site, with the cloaking dropped.   The site beyond Centrale may be cloaked, but that will be in place of any Centrale-provided cloaking
  3. Sometimes, if a sub-web address is forced (e.g., by linking from outside Centrale, as when the URL is typed directly into the browser address window), such as http://orcmid.com/centrale/web/cloaking.htm, the detailed address will become the "cloak" address regardless of any navigation that occurs. 

This is the form of cloaking accommodation that is a practice of Centrale site-construction style.

2. Why Cloaking Failed

First, although cloaking preserved the domain name, it had three key limitations:

  1. The URL used on entry to a page would be the one preserved for all navigation until a new top level is reached and the cloak-applied frame is dropped.
  2. The use of fragment identifiers failed, in that the cloaking service dropped fragment identifiers.  So URLs for fragments (such as into Orcmid Readings) were not usable.
  3. With a little experience blogging, it was also discovered that RSS feeds and other URI-usage that actually fetched material at a given URL were defeated by the presence of the outer, cloaking frame.

Although it was the last situation that demonstrated how cloaking was always a bad idea, it was already very confusing to users.  So, we shall go forth and cloak no more.  If I find another way to preserve domain distinctness on a single anchor site, I will let you know.
