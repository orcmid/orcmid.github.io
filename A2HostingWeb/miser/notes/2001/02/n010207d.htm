Summary

There are casual notions of "algorithm" that are applied in many popular situations, such as when considering that computer programs are algorithms, or are algorithmic. 

There is also rigorous use of "algorithm" as a procedure that has well-defined and confirmable properties.  In this case, algorithm is kin to the idea of "effective procedure" and "effective computability," general notions of being able to solve a class of problems via finite, mechanical means requiring no ingenuity to carry out.  This meaning, from mathematical logic and computer science, is essential to the Miser Project.  We will not be using "algorithm" in any other sense. The use of "algorithm" in this manner is going to be traditional and not so casual as many popular usages.

There are also notions of higher-level algorithms.  Those are excluded here.  There might be future consideration with regard to higher-level arrangements involving the Miser Project.  Those will be addressed separately whenever there is future relevance to advanced situations.

Here, an expression qualifies as the expression of an algorithm if it satisfies some very specific conditions:

  1. It describes operations on particular kinds of objects at a consistent level of abstraction.
  2. It can be demonstrated that carrying out of the operations in accordance with the described procedure will solve the problem in question and it will not provide incorrect solutions.
  3. So long as the operations that the procedure relies upon can be carried out in finite time in a mechanical manner, the procedure always terminates with a solution in finite time using finite resources.

These notions will be illustrated by exhibiting a computer program, a related algorithm description, and then distilling out the essentials of algorithm specification.  Finally, the importance of this understanding for demonstrating characteristics of Miser will be identified.

1. Algorithm Illustrated

To see how much goes into the rigorous expression of an algorithm, we'll now consider, first, a simple computer program.

1.1 A Binary Search Implementation

Here is a procedure expressed in the Python programming language.  It is not critical to understand Python.  The contrast with expression of the algorithm involved is what will be explored. 

def BinSearch(List, item):
    "return the index of item in the already-sorted List"
    "return -1 when item is not in List"
    "based on Donald E. Knuth's Algorithm 6.2.1B"
    "  cleaned up to work with 0-origin lists, tuples, and indexes"
    L = 0                    # Current left-most position
    R = len(List)            # Current right boundary
    while L < R:
        j = L + int((R-L)/2) # A place to try in the 
        if List[j] == item:  #   candidate interval
            return j
        if List[j] < item:   # When no match yet, close off
            L = j + 1        #   the area where item can't be
        else:                #   and continue in the interval
            R = j            #   that remains
    return -1                # Return impossible index if no match

 Notice how much of the above expression consists of comments in "..." and also to the right of "#" marks.  If those were stripped away, we would have nothing but

def BinSearch(List, item):
    L = 0                   
    R = len(List)           
    while L < R:
        j = L + int((R-L)/2)
        if List[j] == item: 
            return j
        if List[j] < item:  
            L = j + 1       
        else:               
            R = j           
    return -1

An experienced software developer with fluency in the Python language could determine what this procedure accomplishes and even identify defects in it, if any.  Many will prefer this form and consider the narrative in the first form distracting and unnecessary.  Although this is the only part of the procedure that is employed by the computer, it is not enough to establish the algorithm and what it accomplishes.  There is too much tacit understanding required if the procedure is to be offered as the expression of an algorithm.

With either form of the procedure text, there are a variety of incidental characteristics that might or might not have been explicitly considered, including the following.

  • It is assumed that len(List), the number of items in the list, is always representable as an exact integer value and there is no risk of List having more elements than the largest-representable integer. 

  • The procedure can operate with any item and List-item data type so long the values are comparable with < and == (equality).

  • BinSearch employs an impossible index (for Python lists) of -1 to report unsuccessful completion of the search.  The implementation assures that this value cannot arise as a successful result.

  • The program variables L, R, and j are private to the implementation and the only operations with them are those that are shown.  

  • The parameters List and item are assumed to be stable and non-volatile, with no alteration occurring to them while BinSearch is being carried out.

  • The operations are clearly stated, and the purpose for the procedure is explained, with annotation of how particular operations narrow in on a solution, if any.  Identification of the particular algorithm and preservation of essential conditions is by appeal to a separate publication [ACP 6.2.1B]

  • The mysterious form "j = L + int((R-L)/2" deals with two edge cases to accomplish what is, mathematically, j = floor((L+R)/2), the largest integer not greater than the average of L and R.  The modification protects against the unlikely case that L+R "overflows" to an incorrect value.

1.2 A Binary Search Algorithm

To see the difference between the statement of an algorithm for binary search and the BinSearch implementation, above, we will use the style of algorithm presentation introduced by Donald Knuth[Knuth1997: 1.1] for our expression of Algorithm 6.2.1B.

Algorithm Bin (Binary search).  Given a list of items K0, K1, ..., KN-1
in non-decreasing order K0 < K1 < ··· < KN-1, 
this algorithm finds a position, if any, having a given item, K.

Bin1. [Initialize.]
Set L to 0; set R to N, the number of items.
(If there are no items, N is 0.)
 
Bin2. [No Luck?]
Let Rem = R - L, the number of eligible items remaining.
If Rem = 0, terminate the search as unsuccessful.
(Each time we return to Bin2, Rem is smaller and we must eventually find a matching item or terminate with all possibilities eliminated.  See Bin4 and Bin5 to confirm that.)
 
Bin3. [Take a Look.]
(L < R and if K is among the eligible items then KL < K < KR-1.)
Set j to floor((L+R)/2).
(j is always an index within one place of the middle of the interval L through R-1.  L < j < R-1 always.)
If Kj < K, go to Bin4.  If K < Kj, go to Bin5.
Otherwise, the search is successful and Kj is a matching item.
 
Bin4. [Look Higher Next.]
(If K is among the eligible items, then Kj+1 < K < KR-1.)
Set L to j+1.  Go to Bin2.
(0 < R - L < Rem.  We have eliminated at least half of the current items as ineligible.)
 
Bin5. [Look Lower Next.]
(If K is among the eligible items, then KL < K < Kj-1.)
Set R to j.  Go to Bin2.
(0 < R - L < Rem.  We have eliminated at least half of the current items as ineligible.)

This statement of the algorithm is contrived so that someone skilled in the analysis of algorithms can independently confirm some important characteristics:

  • The preconditions in which the algorithm is assured to operate are clearly established
  • We rely on  mathematical objects, not computational ones, so there is a generality beyond the data representations that tend to be used in computer implementations.
  • All operations are definite and presumed to be carried out in finite time and space.  This includes use of floor(x) for the largest integer not greater than x.  
  • At the same time, we are not so definite as to fix the operations in a rigid way.  It is presumed that there is or can be made to be such a fixation when the algorithm is embodied in an implementation.
  • No indefinite operations (e.g., reference to KN or K-1) are ever attempted.
  • The algorithm must terminate.
  • If the input conditions are satisfied, the algorithm accurately determines an index for a K-matching item if and only if there is one.  The algorithm will terminate unsuccessfully if an only if there is no K-matching item.
  • One can analyze the performance of the algorithm in terms of the number of times step Bin3 is ever performed.

A software developer skilled at understanding algorithms and producing codes that achieve the particular result using a computer could use the expression of the algorithm as a basis for a programming-language form that accomplishes the algorithm in a particular setting.  The software could be written in a way that the connection to the algorithm is accounted for.   That is how the annotated version of BinSearch arose.  Although there have been some transformations of the operations in the Python version, preservation of the essential algorithm is still evident and can be confirmed by inspection.  In this particular case, there was iteration between the two in order to strengthen the connections as well as establish the contrasts.

Note that the algorithm, being expressed with regard to abstract indexes and indexed lists of values, need not consider limitations of computer representations.  It is important to adjust for that and not blindly make literal implementations that rely on incorrect assumptions about the representation.  In that respect, it is somewhat helpful that the expression of the algorithm does not look like a computer program that could simply be lifted into code in a modern programming language without giving it any thought.

An indispensible quality of the algorithm narrative is the provision of observations about what the state of affairs is asserted to be at any point in the procedure.  There are annotations given in parenthetical statements and  made on the operations as well..  These are essential to expression of the algorithm, although they are not part of the procedure.    The annotations provide assertions about the algorithm that are important in comprehending the achievement of its function and of its efficacy.  The annotations also support confirmation that the recipe given is indeed that of an algorithm for the identified function.  One relies on the annotations in understanding what is supposed to be the case, and then verify that the assertions hold.   It is then clear that, because the assertions do hold, (1) the procedure always terminates for a finite list, (2) a correct index is always returned if the sought value is in the list, and (3) an incorrect index is never returned.
