This HTML file (ARCHTEMP.HTM) is a boilerplate document that may be used to produce a DMA Architecture Proposal in HTML format. This file was derived from George Florentine's ARCHTEMP.RTF, a refinement of the outline developed in the January, 1996, Technical Committee meeting.
The basic approach to using this file is to open it in an editor (such as Microsoft Internet Assistant for Word95), save it under a new file name, and then edit the new version, transforming it into the HTML file that you desire.
All of the "rubric" entries, providing instructions about usage of the boilerplate, such as this very paragraph, are in emphasized (italic) block quotes. When you have completed your editing of the boilerplate, all of the rubric should be deleted.
As you become proficient using this format, you may want to make a copy of ARCHTEMP.HTM that has abbreviated and customized boilerplate that suits your own style and work approach.
Also, the tips for use with Microsoft Internet Assistant for Word95 can be expanded or replace as appropriate to your situation.

4.1 Integration Model

How does your change impact the DMA integration model ? The integration model addresses the following general areas:
  • Service element integration
  • Applications locating information about DMA systems in their environment
  • Installing DMA systems into operating system environments
  • DMA integration with other sw architectures:
    • COM
    • ODMA
    • OLE
    • Internet technologies
      • HTTP
      • HTML
      • MIME
      • Java
    • OMG services

4.2 Query

How does your change impact the DMA query model ? The query model addresses the following general areas:
  • Query construction
  • Query execution
  • Result set management
If you are introducing new objects, do they participate in the query model (ie can they be queried ?)

4.3 Internationalization (I18N)

As of mid-February 1996 the DMA architecture is being reviewed and revised to incorporate support for global environments. Between now and our April meeting submission authors should coordinate with the DMA NLS subcommittee to understand what changes your submission may have on the internationalization of the DMA architecture. General areas to consider are:
  • Character data types will be wide
  • Message strings will be localized
  • DMA systems will be statically typed with their character sets (The nature of this typing, and how static/dynamic it is are still under discussion)
  • Where in the system character set vs. language should be identified

4.7 What requirement does this address?

Whenever possible, trace the submission back to a specific requirements in the user requirements document for DMA. In general, our bias should be to strongly question any architecture change request that can't be linked back to a requirement. If our customer base didn't ask for this feature, why are we providing it ?

5. Reference Materials

We want to insure that consideration has been given to existing research into each of our topic areas. This area may be excluded in a draft submission, but must be completed in a final submission, and must include at least 2 references. These may be specific journals, or existing product documentation, or notes from a verbal/e-mail conversation with a topic expert.

6. Existing Systems and Related Experiences

This is similar to the previous section, but is more focused on existing product implementations. For example, the containment subcommittee might reference a journal arcticle describing electronic filing cabinets in section 5. In this section a list of the features and failings of existing implementations could be presented. This provides reviewers with insights into the state of the existing art in this particular technical area.

7. Compliance & Conformance Considerations

This section will be weak quantitatively because we've excluded compliance and conformance testing from our v1 set of deliverables. However, if your submission introduces any unique or potentially contentious issues around compliance and conformance, address those issues here. For example, if your submission provides for a wide ranging level of conformance, or perhaps a very narrow range of conformance (works or doesn't), describe these features here in qualititative terms. If you have suggestions for how conformance levels could be defined, include them in your submission.

8. Sample Usage

Provide an example of how this feature is used. This can be provided as either a code example, or as descriptive text. If you can't think up a compelling sample usage, you should not be suggesting this enhancement !

9. Glossary / Nomenclature

List new or ambiguous terms you've used in your submission. Commonly used DMA terms such as "document space" or "service provider interface" do not need to be redefined in your submission. However, any new terms, or terms you feel have been ambiguously used prior to your submission should be included in this glossary.

10. Impact on Existing Systems

Describe how this change will impact existing systems that could be integrated into a DMA system. For example, including a containment model would allow repositories that have a containment model to externalize this through their document space. Browsers that support hierarchical navigation could navigate through DMA object hierarchies using the DMA containment model, and display these hierarchies in their Uis.
If you are making a change that will change the behavior of functionality already specified in the DSCD, specify the type of change in this section.

11. Performance Considerations

Describe the performance impact of your suggested changes. Does your change force implementations that have poor performance ? Consider the fact that DMA systems will be implemented in distributed environments.

12. Backwards Compatibility and Legacy Considerations

This section is probably not applicable for the first round of submissions, but will be useful in subsequent revisions to the architecture.

13. Incorporation Into DMA Specifications

Describe the specific changes that should be made to the architecture to incorporate your change. Describe new sections that should be added, additions/deletions to existing sections. Specify if your change includes updates to the objects and interfaces Access databases used to generate the spec.
Here is a very good place for the authorship and update information, including the date and time of last change. (If you are lucky, this can be provided automatically from the editable form that you are using.)
Also provide editor / author identifications and any links for contact or mail to that person.
